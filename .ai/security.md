# Security & Worker Scoping Model

OrderKart features a rigorous security model designed to safeguard customer directories and proprietary transaction ledger information when deploying the app on worker-controlled devices.

## 1. Cryptographic Specifications

To enforce security under completely offline P2P networks, the app leverages standard cryptography:

- **AES-256 Encryption (CBC Mode & PKCS7 Padding)**:
  - Used to encrypt the SQLite database file (`database.enc`) and configuration files inside provisioning ZIP packages.
  - The derived key is generated by calculating the SHA-256 digest of the worker's unique secret key.
  - A cryptographically secure random 16-byte Initialization Vector (IV) is generated for each operation and prepended to the encrypted payload.
- **HMAC-SHA256 Manifest Signatures**:
  - The package manifest is verified using an HMAC-SHA256 digest generated using the worker's secret key.
  - To guarantee identical serialization across diverse Android/iOS devices, the manifest's keys are alphabetically sorted before serialization into a canonical JSON string.
- **Constant-Time Verification**:
  - Manifest verification (`SecurityHelper.verifyManifest`) compares signatures byte-by-byte using bitwise XOR, preventing side-channel cryptographic timing attacks.
- **Salted Owner PIN Hashing**:
  - Numerical owner PIN credentials are salted and hashed locally using SHA-256:
    $$\text{Hash} = \text{SHA256}(\text{pin} + \text{":"} + \text{salt})$$
  - The `salt` is derived from the epoch timestamp in milliseconds when the PIN was configured.
- **Default Activation Code**:
  - Prior to initial PIN setup, the default activation code is validated against a pre-computed SHA-256 hash:
    `460d235c0ac08c373da0a269e57569aeaa50721061ea966758f57eef78e6e946`

## 2. Authentication & Lockout System
To prevent brute-forcing of the owner administration dashboard, the app implements a local lockout schedule:
* **Failed Attempts Track**: Tracked in local `SharedPreferences` keys (`owner_failed_pin_attempts`, `owner_pin_lockout_until`).
* **Lockout Intervals**:
  * **5 Failed Attempts**: Temporary lockout for **30 seconds**.
  * **10 Failed Attempts**: Extended lockout for **5 minutes**.
* **Reset**: Successful authentication resets both failed attempt counters and lockout timestamps.

---

## 3. Scoped Data Partitioning (Owner-to-Worker)
- **Concept**: Workers must only access customers, locations, and schedules that are explicitly assigned to them by the Store Owner.
- **Implementation**:
  - Whenever the Store Owner generates a provisioning package (`WorkerPackageService.generateWorkerProvisioningPackage`) or hosts a local P2P synchronization socket, the app queries `worker_assignments` for the target `worker_id`.
  - It filters all customer, order, payment, and note queries to select rows matching the assigned `street_id` or `location_id`.
  - The database file cloned into the ZIP is pruned, containing **only** this scoped subset.

---

## 4. Worker-to-Worker Transfer Block
- **Rule**: Workers are strictly prohibited from sharing database exports or local sync sockets with each other.
- **Enforcement**:
  - During import validation (`PackageValidator.validatePackage`) and database merge operations (`DatabaseHelper.mergeDatabaseFromJson`), the app reads the `app_mode` and the manifest's `generated_by_worker_id`.
  - If both source and destination are workers, the operation is blocked and throws an exception, preventing unauthorized horizontal propagation.
